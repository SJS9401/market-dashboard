### 파일 1: feedback_earnings_project_integration.md

```markdown
---
name: 실적 3종 모드 ↔ quarterly-review 연계 재설계
description: earnings-preview / earnings-review / earnings-in-depth 3종 user skill을 quarterly-review 시스템과 자동 연계되도록 재설계한 통합 룰 (출력 위치·메타데이터·tier 강화·연계 흐름).
type: feedback
---

earnings-preview / earnings-review / earnings-in-depth 3종 모드는 quarterly-review 3-stage 워크플로우와 자동 연계되도록 설계되어야 한다.

**Why:** 종전에는 3종 모드 산출물이 .docx로 흩어져 저장되어, 분기 단위로 종목·섹터를 누적 추적하기 어려웠다. quarterly-review의 [분기 섹터 분석 모드](Stage 2)가 review·in-depth 산출물을 자동 로드해 통합 분석하려면, 산출물 위치·파일명·메타데이터가 표준화되어야 한다.

**How to apply:** 3종 모드 SKILL.md에 다음 6개 룰을 동일하게 반영한다.

(1) 출력 파일 위치 통일
→ `WORK_DIR/earnings-{mode}/{YYYY-Q}_{종목}_{suffix}.md`
→ 예: `earnings-in-depth/2026-Q1_SK하이닉스_인뎁스.md`
→ suffix: 프리뷰 / 리뷰 / 인뎁스
→ mode: preview / review / in-depth
→ .docx가 아닌 .md 출력 (quarterly-review가 자동 로드 가능하도록)

(2) 본문 위계 5단계 적용
→ 별도 메모리(feedback_daily_issue_hierarchy.md) 참조

(3) F&G 한글 표기 룰
→ 별도 메모리(feedback_fng_korean_label.md) 참조

(4) 메타데이터 헤더 (마크다운 인용블록 ">")
→ 파일 본문 최상단에 6줄 작성:
```
> 모드: [실적 프리뷰 / 실적 리뷰 / 실적 인뎁스 분석]
> 종목: [종목명] ([티커])
> 섹터: [섹터명 — 워치리스트 [분기 주도 섹터]와 일치]
> 분기: [YYYY-Q]
> 발표일: YYYY-MM-DD
> 작성 시각: YYYY-MM-DD HH:MM KST
```
→ YAML frontmatter(---) 아님. 마크다운 인용블록(>) 사용.

(5) quarterly-review 시스템과의 연계 흐름
→ 별도 메모리(feedback_quarterly_review_system.md) 참조
→ Stage 2가 **review .md + in-depth .md 둘 다 자동 로드**해서 통합

(6) tier 강화 룰 (3-tier 시스템 — feedback_sector_tier_system.md 참조)
→ **T1·T2 섹터의 종목**: preview → review → in-depth 풀 사이클
→ **T3 섹터의 종목**: preview + review만 (in-depth는 시그널 감지 시만)
→ 발표가 미뤄져도 preview는 매번 갱신 (모든 tier)
→ 분기 단위 누적 추적 (T1·T2 풀, T3 가벼움)
```

### 파일 2: feedback_quarterly_review_system.md

```markdown
---
name: quarterly-review 3-stage 워크플로우 전체 그림
description: quarterly-review 시스템의 Stage 1~3 흐름과 earnings 3종 모드와의 자동 연계 조건 (cron 스케줄·섹터 매칭·누락 안내·tier별 깊이 차등).
type: feedback
---

quarterly-review 시스템은 3-stage 워크플로우로 구성되며, earnings 3종 모드 산출물을 자동으로 로드해 섹터·분기 단위 통합 분석을 수행한다.

**Why:** 분기 실적 시즌이 끝날 때마다 종목별 분석을 수동으로 모아서 섹터 단위로 재가공하는 작업이 반복적이었다. earnings review·in-depth 산출물을 표준 위치/메타데이터로 저장하면, quarterly-review가 자동으로 섹터별 종목을 모아 분기 주도 섹터를 판정할 수 있다.

**How to apply:**

(1) Stage 1 — 종목별 분석 모드들
→ [실적 프리뷰 모드]: `WORK_DIR/earnings-preview/{YYYY-Q}_{종목}_프리뷰.md`
→ [실적 리뷰 모드]: `WORK_DIR/earnings-review/{YYYY-Q}_{종목}_리뷰.md`
→ [실적 인뎁스 분석 모드]: `WORK_DIR/earnings-in-depth/{YYYY-Q}_{종목}_인뎁스.md`

(2) Stage 2 — [분기 섹터 분석 모드]
→ 섹터 단위 통합 분석. **tier별 깊이 차등** (feedback_sector_tier_system.md 참조)
  - **섹터 T1**: 풀 깊이 (가설 형성·논쟁 분석·통합 시사점)
  - **섹터 T2**: 가벼운 분석 (사이클 위치·핵심 시그널 위주)
  - **섹터 T3**: 시그널 감지 (Beat/Miss 패턴, 컨센 변화 → T2 승격 후보 검출)
→ **review .md + in-depth .md 둘 다 자동 로드** (review = 발표 직후 정리, in-depth = 시장 반응 후 쟁점)
→ preview는 발표 전 자료라 보조 참조용
→ 워치리스트(`WORK_DIR/watchlist.md`)를 자동 로드해 tier별 처리
→ 메타데이터 헤더의 [섹터] 필드로 워치리스트와 매칭
→ 누락 종목이 있으면 사용자에게 "[종목] {모드} .md 미존재" 안내

(3) Stage 3 — quarterly-review 자동 스케줄러
→ cron: `0 8 16,31 3,5,8,11 *`
→ 본격 실행 일자: 5/16, 8/16, 11/16, 3/31 오전 8시 KST

**자동화 작동 조건:**
- 모든 review·in-depth .md가 표준 위치에 저장
- 메타데이터 헤더 [섹터] 필드가 워치리스트의 명칭과 정확히 일치
- 분기 식별자(YYYY-Q) 일관 사용

**review와 in-depth의 역할 구분 (Stage 2 통합 시점):**
→ review: 실적 vs 컨센·가이던스 갭, 경영진 코멘터리, BU별 마진/볼륨, 사이클 위치, **셀사이드 컨센 변화 (5단계 분류)**
→ in-depth: 시장 반응 후 논쟁 쟁점, 양면 분석, 모니터링 변수
→ Stage 2는 둘을 종합하여 섹터 단위 사이클·논쟁·시장 시각 통합본 산출
```

### 파일 3: feedback_daily_issue_hierarchy.md

```markdown
---
name: 본문 위계 5단계 표기 룰 (일일 이슈 + 실적 분석 공용)
description: 분석 본문 작성 시 ## / ① / (1) / (1-1) / → 5단계 위계를 일관되게 사용하는 룰.
type: feedback
---

분석 본문은 항상 5단계 위계로 작성한다. 단계 마커를 임의로 변경하지 않는다.

**Why:** 위계 마커가 들쑥날쑥하면 같은 분기 내 종목·섹터 간 흐름 비교가 어렵다. quarterly-review가 review·in-depth .md를 자동 파싱해 섹터 단위로 통합할 때도 위계가 일관되어야 정확하게 추출된다.

**How to apply:** 다음 5단계 마커를 그대로 사용한다.

(1) 1단계 — 대주제: `## 헤더`
(2) 2단계 — 중주제: `①` `②` `③` (원형 숫자)
(3) 3단계 — 소주제: `(1)` `(2)` `(3)` (괄호 숫자)
(4) 4단계 — 세분화: `(1-1)` `(1-2)` (하이픈 연결)
(5) 5단계 — 내용: `→` (화살표로 시작)

**금지:**
→ 5단계 이상 위계 (예: (1-1-1))
→ ① 대신 1., A., 가. 등 다른 마커
→ → 대신 -, *, • 사용 (잎 노드는 항상 →)

**적용 범위:** 실적 프리뷰/리뷰/인뎁스 분석 .md 본문, 일일 이슈 분석, 분기 섹터 분석 등 모든 .md 산출물.
```

name: F&G(Fear & Greed) 한글 공식 영역명 표기 룰

### 파일 5: feedback_chart_embed_rule.md

```markdown
---
name: 분석 산출물 차트·이미지 임베드 룰 (matplotlib + 한글 폰트)
description: 실적 분석 .md 산출물에 차트를 임베드할 때의 표준 — Flat 저장, 한글 폰트 자동 등록, 미국/한국 차트 컨벤션.
type: feedback
---

**Why:** quarterly-review가 .md를 텍스트 파싱하므로 차트는 자동화에 영향 없음(이미지 무시). 사람이 .md를 읽으면서 토론·수정하는 과정에서 시각 자료가 핵심이라 차트 표준화가 필요.

**How to apply:**

(1) 저장 위치 & 명명 (Flat 구조)
→ 위치: .md 파일과 같은 폴더 (예: `WORK_DIR/earnings-preview/`)
→ 명명: `{YYYY-Q}_{종목}_{차트설명}.png`
→ 예: `earnings-preview/2026-Q1_NVDA_revenue_trend.png`

(2) 한글 폰트 자동 등록 코드 패턴

```python
import matplotlib
matplotlib.use('Agg')
import matplotlib.pyplot as plt
import matplotlib.font_manager as fm
import glob

font_files = glob.glob('/sessions/{session}/mnt/2 실적 프리뷰-리뷰/.fonts/*.ttf')
if font_files:
    fm.fontManager.addfont(font_files[0])
    font_prop = fm.FontProperties(fname=font_files[0])
    plt.rcParams['font.family'] = font_prop.get_name()
plt.rcParams['axes.unicode_minus'] = False
```

→ 폰트 파일 없을 시: 영문 차트 fallback

(3) 차트 컨벤션 — 미국/한국 분기

**미국 기업:** 제목 한글 / 축 라벨 영문 / 분기 표기 회사 회계 기준 (`FY26 Q1`)
**한국 기업:** 제목 한글 / 축 라벨 한글 / 분기 표기 `1Q25`, `2Q26(E)`

(4) 시각 표준
→ 듀얼축: 매출(바) + 마진(GPM/OPM)(선)
→ 매출 색상: 미국 청색(`#1f77b4`)·한국 녹색(`#2ca02c`)
→ 마진 적색(`#d62728`) 통일
→ (전망)·(추정): 매출 바 연한 색 + 라벨 `(G)`/`(E)`
→ DPI: 100~150

(5) .md 본문 임베드
→ 상대 경로 사용
→ `![대체텍스트](파일명.png)`
→ 차트 직전·직후 캡션·출처를 5단계 마커(`→`)로 명시

**적용 범위:** earnings-preview / earnings-review / earnings-in-depth 3종 모드 산출물 + quarterly-review 통합 리포트.
```

### 파일 6: feedback_sector_memory_structure.md

```markdown
---
name: 섹터별 세션 운영 + 섹터 지식 메모리 구조
description: 실적 분석 프로젝트를 섹터별 세션으로 운영하는 워크플로우 (나→다 전환), 섹터 지식 메모리 3종 (가설·변수·미해결 논점)의 저장 위치·구조·갱신 시점.
type: feedback
---

실적 분석 프로젝트는 **섹터별 세션** 단위로 운영한다.

**Why:** quarterly-review Stage 2가 섹터 단위로 review·in-depth를 통합 분석하므로, 같은 세션에서 한 섹터의 모든 종목을 다루면 산업 컨텍스트가 누적 활용되고 Stage 2 호출이 자연스럽다.

**How to apply:**

(1) 세션 운영 모드: 나 → 다 전환

(1-1) 나 단계 (시작) — 섹터 단일 세션 누적
→ 한 섹터 = 한 세션 (예: "반도체 실적 분석", "전력 실적 분석")
→ 모든 종목 × 모든 분기의 preview/review/in-depth 누적
→ 분기 종료 시 quarterly-review Stage 2까지 같은 세션에서

(1-2) 다 단계 (전환 후) — 분기 단위 새 세션
→ 비대화 트리거 발동 시 다음 분기부터 새 세션
→ 새 세션은 워크스페이스 `sectors/{섹터명}/` 폴더 첨부로 컨텍스트 복원

(1-3) 전환 트리거 (둘 중 먼저)
→ 분기 수: 4분기 누적
→ 체감: BT가 응답 속도 저하·깊이 얕아짐 인지 시

(2) 섹터 지식 메모리 저장 위치
→ 워크스페이스 `WORK_DIR/sectors/{섹터명}/`
→ Cowork `.auto-memory/`가 아님
→ 7개 프로젝트 룰 메모리는 `.auto-memory/` 그대로 유지

(3) 섹터 지식 메모리 3종

(3-1) `working_hypotheses.md` — 섹터 핵심 가설 (1~3개)
(3-2) `tracking_variables.md` — 추적 변수 표
(3-3) `unresolved_in_depth_points.md` — 미해결 in-depth 논점

(4) 갱신 시점 — 분기 종료 시 (Stage 2 직후)

(5) 다 전환 시 새 세션 셋업 절차
→ `sectors/{섹터명}/` 폴더 첨부
→ 첫 메시지: "{섹터명} {YYYY-Q} 분석 시작. 첨부한 섹터 메모리 기반으로 컨텍스트 복원."

(6) 새 섹터 시작 시 템플릿 사용 (자동 처리)
→ `sectors/.template/`에 빈 템플릿 3종 보관
→ **자동 생성 룰**: 새 섹터의 첫 분석 모드 호출 시 (예: `[실적 프리뷰 모드] HD현대일렉트릭`):
  (6-1) 워치리스트(`watchlist.md`)에서 종목 → 섹터 매핑 확인
  (6-2) `sectors/{섹터명}/` 폴더 존재 여부 확인
  (6-3) 없으면 `.template/` 복사 → `[섹터명]` 플레이스홀더를 실제 섹터명으로 치환 → 폴더 생성

### 파일 7: feedback_sector_tier_system.md

```markdown
---
name: 섹터 3-tier 시스템 (T1·T2·T3 분류)
description: 분기별 분석 대상 섹터를 메인(T1)·주도 후보(T2)·트래킹(T3) 3-tier로 분류하여 작업 깊이와 Stage 2 처리를 차등화하는 룰.
type: feedback
---

분기별 분석 대상 섹터를 3-tier로 분류한다.

**명칭 컨벤션 (중요):** 이 시스템의 T1·T2·T3는 항상 **"섹터 T1·T2·T3"**로 호칭한다 (예: "섹터 T1은 반도체·소부장·전력·배터리"). BT가 별도로 운영하는 **이슈 팔로업 워치리스트의 T1-T3**과 구분하기 위함이다. 단독 "T1"은 모호하므로 사용 금지.

**Why:** BT는 한 분기에 10~15개 섹터를 추적하는데, 모든 섹터를 동일한 깊이로 분석하면 ROI가 떨어진다.

**How to apply:**

(1) 3-tier 정의
→ **섹터 T1 — 메인** (3~4개): 풀 사이클 + Stage 2 풀 깊이
→ **섹터 T2 — 주도 후보** (3~5개): 풀 사이클 + Stage 2 가벼운 분석
→ **섹터 T3 — 트래킹** (3~5개): preview+review만 + Stage 2 시그널 감지

(2) tier 분류 주체
→ **분기 시작 시 BT가 직접 분류**
→ 워치리스트 파일(`WORK_DIR/watchlist.md`)에 명시

(3) tier 동적 조정
→ T3 → T2 승격: 강한 시그널 발견 시
→ T2 → T1 승격: 분기 주도 섹터로 부각 시
→ 승격·강등은 BT 결정. 시스템은 시그널을 알림으로 제안

(4) T3 시그널 감지 룰 (Stage 2 시점)
→ **Beat/Miss 패턴**: 섹터 내 다수 종목 컨센 +5% 이상 Beat → T2 후보
→ **셀사이드 컨센 변화** (review 항목 6 데이터): 5단계 분포 1단계 이상 이동 多 → T2 후보
→ **사이클 시그널**: 섹터 가격 추세 전환, 글로벌 피어 동조 호조
→ Stage 2 출력에 "T2 승격 후보: [섹터명]" 명시

(5) 워치리스트 구조 (`WORK_DIR/watchlist.md`)
→ 분기별 in-place 갱신
→ 섹터 T1 / T2 / T3 분류 + 섹터별 종목 리스트
→ Stage 2가 자동 로드해 [분기 주도 섹터] = T1+T2로 매칭

(6) 기존 "T1 종목 강화 룰" 재정의
→ **T1·T2 섹터의 종목 = 풀 사이클** (preview/review/in-depth)
→ T3 섹터의 종목 = preview+review만 (in-depth는 시그널 감지 시)
→ 분기 누적 추적은 모든 tier에 적용 (T3는 가벼움)
```
